# ADR 0001: Reusable Workflow は main 参照と明示的な命名を採用する

## ステータス

採用。

## 背景

このリポジトリは4本の Reusable Workflow を個人管理の複数リポジトリへ提供している。`@v1` は `main` の修正を利用側へ届けるためにタグ更新を必要とし、更新漏れにより修正済み Workflow が配布されない障害が発生した。既存の `-wc` という接尾辞にも正式な定義がなく、呼び出される側の Workflow であることを読み取れない。

## 決定

- 配布先は Reusable Workflow を `@main` で参照する。
- 呼び出される側は `.github/workflows/<用途名>-reusable.yml`、呼び出し側は `<用途名>.yml` とする。
- 個別仕様は `docs/reusable-workflows/<用途名>.md`、横断設計は `docs/reusable-workflows.md` に置く。
- Reusable Workflow の変更はテストと代表リポジトリでの確認後に `main` へマージする。
- 既存の `v1` と `v1.0.0` は配布先の移行完了まで残し、参照がなくなった後に削除する。

## 理由

個人管理の共有基盤では、利用側を互換性境界ごとに固定する必要性より、修正を確実に全利用側へ届けることを優先する。`-reusable` は GitHub Actions の用語と一致し、略語を解釈せずに役割を理解できる。

## 結果

- タグの作成・付け替え・検証は不要になる。
- `main` の変更は全利用リポジトリの次回実行へ反映されるため、共有 Workflow の PR では検証が必須になる。
- 外部提供や複数チームでの独立したリリース管理が必要になった場合は、メジャータグまたはコミット SHA 参照を再検討する。

## 検討した代替案

- `@v1` と不変のパッチタグを維持する。互換性境界には有用だが、4本の Workflow を個人で管理する現在の規模では更新漏れの運用コストが上回るため採用しない。
- すべてをコミット SHA で固定する。再現性は最も高いが、各利用リポジトリでの更新作業が必要になるため採用しない。
